# Quartal Loom — 要件定義

> 正本。計画ハブ(Dorothy `work/music/theory-circle-app/_epic.md`)からリンクされる。
> ここは「何を作るか」の SSOT。実装の詳細判断はコード＋テストが正本。

## 1. 目的
4度圏を軸に、**任意のキー/スケールでオンスケール領域とテンションを動的表示**する。
現行の静的な4度圏表(Aエオリアン固定・テンション目視)の2欠点——①転調時の境界が読めない
②テンションの正しさを確認できない——を、整数モデルの計算で解く。

## 2. スコープ（今回 = 動的表示 MVP）
### やる
- key × scale を選ぶと、4度圏上のオンスケール/アウトが動的に変化して見える。
- 各コードの可用テンションを堆積ルールから計算して表示（現行画像の動的版）。
- スケール/コードを**データとして**持つ（組込ライブラリ）。
### やらない（将来拡張。ただし壊さないよう器だけ今設計）
- 発音・MIDI 出力。
- コード進行の入力・保存・移調。
- スケール/コードの**追加・編集 UI**（＝ライブラリ管理UI）。← §5 参照。
- VST 化 / 物理楽器移植。

## 3. 移調の定義（用語を分離）
「移調」は2つの別物。混同禁止。
1. **表示キーの移動（今回の中核操作）**: どのキーを"ホーム"にするか。オンスケール領域が動く。
   - **UX 決定(2026-07-07)**: **円盤ごと回転させ、ホーム(選択キー)を常に最上部に固定**する。
     円の各扇形＝音は不変配置のまま、円全体を回してホームを上へ持ってくる。
   - コア側の実装: `Scale { root, mask }` の `root` を変えるだけ（マスク不変＝平行移動性）。
     回転は純粋に**表示アダプタの責務**。コアは「どの音がオンか」を返すのみ。
2. **保存済みコード/進行の移調**: 別キーへ移す。→ **今回対象外**（進行仮組み側の機能）。

## 4. データモデル（コア = 純粋理論）
- 音 = ピッチクラス `u8`(mod 12)。スケール/コードの「形」= 12bit `IntervalMask`。
- **スケール** = ルート＋マスク。組込 = 教会旋法7種（実装済み）。順次追加。
- **コード** = 「**ピッチクラスの任意集合＋役割ラベル**」のゆるい器。
  機能和声の型に決め打ちしない（4度堆積・Blackadder 等の異形を将来受けるため）。
  - 例: `maj7 = intervals[0,4,7,11]`, 役割 `{0:R, 4:M3, 7:P5, 11:M7}`。
  - テンション = ルート/短長3度からの**完全5度堆積**を関数生成（M2 実装済み）。§4.1 参照。

### 4.1 テンションモデル（5度堆積・2026-07-07 本人確定）
テンションは「完全5度の積み上げ」で導出する。root は major/minor 共通、3度から上が質を決める。
- **共通 root 堆積**（4音）: root から完全5度を4つ = `R, P5, 9, 13`（A なら A E B F#）。
- **メジャー堆積**: 上記 ＋ **M3 からの5度堆積**（3音）= `M3, M7, #11`（A なら C# G# D#）。
  合体 = `A C# E G# B D# F#` = **A リディアン**（5度堆積が natural 11 の avoid を自動回避）。
- **マイナー堆積**: 上記共通 ＋ **m3 からの5度堆積**（3音）= `m3, m7, 11`（A なら C G D）。
  合体 = `A C E G B D F#` = **A ドリアン**。
- コア API: `tensions(root, quality) -> [TensionTone; 7]`。各 tone は pitch・度数ラベル・
  source（共通root堆積 / 質を決める3度堆積）を持つ。

### 4.2 テンション堆積の2スイッチ構造（2026-07-07 改訂）
堆積は **Root** と **3rd** の2スイッチで組む（旧 Major/Minor トグルを一般化）。
- **Root**: `off / 4th / 5th`。
  - `off` → R のみ。
  - `4th` → **純クォータル7音**: R + 完全4度×6（R,q4,q7,q10,q13,q16,q19）。**3rd は必ず off**（M3/m3 無効化）。
    例(R=B): R B, q4 E, q7 A, q10 D, q13 G, q16 C, q19 F。
  - `5th` → R から5度を4回積む（R,5,9,13）。3rd 堆積と併用。
- **3rd**(5th時のみ有効): `off / M3 / m3`。その3度から**5度**で3回積む（M3→M3,M7,#11 / m3→m3,m7,11）。
  - `off` → 3rd 堆積なし。Root=off + 3rd 選択時は R＋3度の2音のみ。
- 既定 = Root:5th / 3rd:M3（＝従来 Major＝リディアン）。5th/m3＝従来 Minor＝ドリアン。
- **実装メモ**: 堆積の音生成(step 積み)は表示アダプタ(web)で行う（`(root+k*step)%12` の算術）。
  core の `tensions()` は正準な5度堆積(Major=Lydian/Minor=Dorian)のリファレンス＋テストとして残す。
  将来 VST 化する際は buildStack 相当を core へ引き上げて SSOT を一本化する（現状は既知の簡略化）。

### 4.3 表示の軸（独立・表示アダプタ責務）
1. **どのテンションを描くか** = §4.2 の2スイッチ。
2. **彩度 = オンスケール判定**: 各テンション音が選択スケールに含まれれば有彩(マルーン)、外れれば無彩。
3. **背景の明度 = TSDロール群**（時計位置 mod3: 12/3/6/9=最暗 > 1/4/7/10=暗 > 2/5/8/11=中）。
4. **操作**: セクタclick=そのルートへ最短回転(ease)、中央=スケール選択、上部=2スイッチ。
   home変更・スケール/堆積変更・フォントサイズ等、動的に変わる見た目は easeInOutCubic で補間。
5. 円盤回転・選択・トグルは表示側。コアは「集合」「オンか否か」を返すのみ。

## 5. ユーザー追加のための「形式」設計（今回はモデルのみ／UIは作らない）
> 決定(2026-07-07): 将来のライブラリ管理UIを念頭に、**データ形式だけ先に確定**する。
> 追加/編集/削除の UI は作らない。組込ライブラリを読む経路だけ用意する。

- **ライブラリ形式 = JSON**（人が手編集でき、将来UIが読み書きする器）。スケールもコードも同型。
- **共通スキーマ（案）**:
  ```json
  {
    "kind": "scale" | "chord",
    "id": "aeolian",           // 一意キー(参照・移調で不変)
    "name": "Aeolian",         // 表示名(略称禁止=フル表記)
    "intervals": [0,2,3,5,7,8,10],   // ルートからの半音。これがマスクの素
    "roles": { "0": "R", "3": "m3" },// chord のみ任意。度数→役割ラベル
    "builtin": true,           // 組込=true / ユーザー追加=false
    "category": "church-mode"  // 任意。将来の一覧グルーピング用
  }
  ```
- **移調の一貫性**: どのエントリも `intervals`(形)は不変で、`root` の付け替えだけで移調が成立する。
  → ユーザー追加スケール/コードも組込と全く同じ移調規則に自動的に従う（特別扱い不要）。
- **層の分離**: JSON パース・永続化は**アダプタ層(web/wasm)の責務**。コア(`no_std`)は JSON を知らない。
  コアは `intervals` から `IntervalMask` を作る入口だけ提供する。
- **拡張余地（今回は実装しない、設計に反しないことだけ担保）**:
  永続先は web MVP で localStorage → 将来ファイル/クラウド。UI は builtin を読み取り専用、
  user エントリを CRUD する管理画面へ育てられる形にしておく。
